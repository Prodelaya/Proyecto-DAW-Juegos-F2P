## RESUMEN EJECUTIVO
Extensión recomendada: 1 página. Debe explicar el proyecto a alguien que no lo conoce.
Incluye problema, solución, tecnologías, resultados y estado (MVP/completo).

**Plantilla (rellenable)**
> Este proyecto desarrolla [TIPO DE APLICACIÓN] para [USUARIO OBJETIVO] con el fin de resolver [PROBLEMA]. La solución ofrece [3 FUNCIONALIDADES CLAVE] y se ha implementado usando [TECNOLOGÍAS]. El sistema se compone de [FRONT/BACK/API/BD], incorpora [SEGURIDAD], y se ha validado mediante [PRUEBAS]. El resultado final es [URL/ENTREGA], con posibilidad de mejora futura en [2 MEJORAS].

**Ejemplo completo (DAW)**
> Este proyecto desarrolla una aplicación web de gestión de reservas para un centro deportivo, dirigida a clientes y administradores. El problema principal es la dependencia de hojas de cálculo y mensajes para coordinar plazas, horarios y pagos, lo que provoca errores y sobreventa. La solución permite: (1) registro e inicio de sesión, (2) consulta y reserva de clases con control de aforo, y (3) panel de administración para crear sesiones y gestionar usuarios. Se ha implementado con una API REST en Node.js (Express), base de datos MySQL y un frontend en React. Incluye autenticación JWT, control de roles y validación de entradas. Se han realizado pruebas funcionales y de validación de formularios. El sistema se ha desplegado en un VPS con Nginx y HTTPS. Como mejoras futuras se plantea integrar pasarela de pago y notificaciones por correo.

## 8. SEGURIDAD Y PROTECCIÓN DE DATOS (RGPD básico)
No hace falta una tesis legal, pero sí demostrar buenas prácticas y conciencia de protección de datos.

### 8.1 Seguridad técnica mínima
* Contraseñas hasheadas (bcrypt/argon2) y nunca en texto plano.
* Validación en backend (no confiar en el frontend).
* Prevención XSS: escape/sanitización en vistas + headers (si aplica).
* Prevención SQLi: consultas parametrizadas/ORM.
* Control de roles y rutas protegidas.
* Gestión de errores sin filtrar datos sensibles (no devolver stack traces en producción).

### 8.2 Protección de datos (contenido sugerido)
* Datos personales tratados (ej. email, nombre, historial de reservas).
* Finalidad del tratamiento (gestión de reservas).
* Medidas aplicadas: minimización, contraseñas hasheadas, HTTPS, control de acceso.
* Derechos del usuario: acceso/rectificación/supresión (explicar cómo se gestionaría).
* Política de retención (ej. borrar cuentas inactivas tras X tiempo, si aplica).

**Ejemplo de redacción**
> La aplicación trata datos identificativos (correo electrónico) y datos de uso (reservas). La finalidad es gestionar el servicio de reservas. Se minimizan los datos solicitando únicamente los necesarios. Las contraseñas se almacenan mediante hash seguro y el acceso se controla por roles. En producción se utiliza HTTPS. El usuario podría solicitar la eliminación de su cuenta, lo que implicaría anonimizar o borrar sus datos según la política establecida.
